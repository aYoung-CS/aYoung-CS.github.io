题目来源：hws2021线上赛

STM

bin文件放到ida里，Processor typr选ARM 小端
Processor options选ARMv7-M
ROM开始地址填0x8000000（固定的开始地址），size会自动计算好

stm32初始化方法：从最低地址找中断向量表，中断向量表存着reset的中断处理函数，会根据这里，也即0x0000 0004指示的地址开始执行代码

进ida后，看到第二个地址指向0x8000101，最后一个1代表thumb模式

跳过来，在0x8000100处单走一个c键，识别出了很多函数和指令

然后就能找到main了
逻辑很简单就是异或加密一下

BlinkBlink

binwalk -Me uImage解固件
getinfo.js有很多接口
goahead中有字符串set_qos_cfg，goahead是一个嵌入式web服务器
在程序gohead中找引用能找到其中定义的接口处

其中有一个set_cmd

grep -r xxxx目录下搜索字符串，找一下bs_SetCmd函数，在libshare-0.0.26.so里

看到取键cmd的值执行popen

最后就直接传命令过去xxx:xx/goform/set_cmd="??"即可

没环境也不知道咋用给的文件系统启动，只好看大佬的wp意会一下了

easybios

可以直接用给的命令启动

qemu-system-x86_64 -bios easybios

进来可以看到提示，使用getflag获得flag。所以应该是需要找到逻辑并逆向

首先用binwalk解文件得到一个80A8的东西，从binwalk的输出也能看到里面有PE文件。直接拉进IDA和010

先用010搜索字符串Wrong!，定位到输入flag错误输出的字符串位置

对应地址37F1B8，然后IDA跳这个地址，发现就是字符串存储的地址。交叉引用找到代码，在sub_33BBB3里

猜一手，v13应该就是输入的字符串，sub_35C12B判断长度为32。注意这里的编码是UTF-16LE。对于涉及到的普通可见字符一个字节就能表示了，另一个字节会是0。这在010中查看字符串二进制的时候也能发现

UTF-16LE Sixteen-bit UCS Transformation Format, little-endian byte order
意思就是两个字节表示一个字符

接着来一个32次的循环，v7跟v13是一个东西，判断了一下输入字符串每个字符都必须是十六进制小写的符号，否则就跳wrong

然后sub_35C59B不知道是啥，不过里面一些string的串看起来像是什么系统函数。再往下看到经过了sub_312169之后有一个逐字节判断，byte_37FA54已经存好了密文。
注意这里sub_312169没有识别出来参数，实际上查看汇编是因为和上一个函数连续了两个call


而进到sub_35C59B中，其开头

返回前

意思就是这俩函数在调用的时候有几个寄存器是一样的。然而这里的调用规则不是很懂，总之就是猜后面那个函数参数也用到了输入字符串v7，并且把加密结果存在v17，这样一来和下面那个比较就说得通了。

加密函数

就是个异或，仔细看发现前面一堆有的没的就是为了得到最后异或的东西，啥也不用管直接抄下来加密过程即可，异或换一下位子得flag

最后还要注意因为用%02x对齐打印出来，补一个0，否则打出来flag比加密前少了一个数字

easymsg

先逆一手协议


开头六个字节是HwsDDW

这里可以看到(*(a1 + 6) << 8) + *(a1 + 7)即第七第八字节表示长度，不大于0x1000

接下来计算crc，报文中传入的crc四字节


显然位置是*(a1+10)~*(a1+13)

sub_E80在计算crc，其中的a1是报文+14后的字符串，a2是长度字段的值


可以看到都给好计算方法了，所以发的时候只需要带进去算一下然后放到报文对应位置就好

接着进入功能函数了，sub_15E8((*(a1 + 8) << 8) + *(a1 + 9), v7);。根据函数里的内容可以知道(*(a1 + 8) << 8) + *(a1 + 9)对应选项，传入的v7是a1+14开始的字符串，是真正的内容，另外里面需要有冒号

至此协议格式已经完整了

功能函数sub_15E8，其中a1对应选项，a2对应内容


有用的就是258对应的sub_1504

看到几个选项


另外发现传参的时候都是命令长度+1，也就是说这里的格式是command:content

还有一个需要注意的点在于注意上面协议从报文取数字时，例如长度为(*(a1 + 6) << 8) + *(a1 + 7)、crc为(*(a1 + 12) << 8) + (*(a1 + 10) << 24) + (*(a1 + 11) << 16) + *(a1 + 13)，意思就是其中的数字是根据正常顺序直接取出然后移位拼出来的，也就是说这里的顺序其实是大端序，在发送数据的时候需要注意
然后长度也不能太短，否则两字节长度会出现空字节，程序使用strcpy复制故会被截断

然后就发现了神奇的事情，根本看不到flag字样

a1是输入的字符串，使用memcpy复制了0x100字节到v6，然后送去检查的只有v6也就是前0x100个字节，而前面我们为了避免空字符截断要控制长度大于0x100，所以只要flag在0x100之后就不会被检查到了，属于过滤了个寂寞。

PowerPC架构

第一次接触，先看这篇稍微学一下https://4f-kira.github.io/2019/06/09/powerpc/#PowerPC

IDA好像无法反编译，而ghidra也没看到什么有用的东西。不懂网上别人开出来的为什么那么美丽

直接测就能看到其实是个栈溢出，且看起来保护也关了。有读写段，可以直接写shellcode

首先测一下缓冲区长度，直接莽然后就能控制lr地址了

然后就是写shellcode了，看大佬的wp是直接到网上找了现成的http://shell-storm.org/shellcode/files/shellcode-86.php
然而我打不通啊=-=，然后找到了大概是这种题（ppc pwn）的原版，国外某个题的wphttps://ctf.harrisongreen.me/2019/utctf/ppowerencryption/

看到他这下面手写的shellcode，直接powerpc-linux-gnu-gcc编译出来然后放进ida里看看，这最后系统调用的sc指令就对应的是0x44000002啊，然后把shellstorm那份shellcode的最后改成0x44000002就通了。。属于不是很理解为什么

老外说了:


于是尝试自己仿照着手写一份。因为ppc也属于精简指令集，指令长度固定四字节，qemu模拟的好像一般地址都不带变的，所以这里赋值r3考虑直接通过计算得到字符串位于栈上的地址。另外编译的时候说是赋值范围要在0xffffffffffff8000到0x0000000000007fff，无法一步到位。那就两步加法就行。

shellcode如下，发的时候后面再跟上/bin/sh，其实也就是清空几个寄存器，r0赋值成execve的系统调用号，r3作为函数第一个参数让它指向binsh即可，最后sc就是系统调用
